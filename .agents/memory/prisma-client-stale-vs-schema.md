---
name: chg-rehab Prisma client can be stale vs committed schema
description: tsc may pass/fail misleadingly because node_modules @prisma/client drifts from apps/chg-rehab/prisma/schema.prisma
---

The generated Prisma client in `node_modules/@prisma/client` can drift from the
committed `apps/chg-rehab/prisma/schema.prisma`. When debugging chg-rehab `tsc`
errors about Prisma fields/models, **regenerate first** before trusting results:
`./node_modules/.bin/prisma generate --schema=apps/chg-rehab/prisma/schema.prisma`.

**Why:** A stale client masks schema drift. Symptom seen: app code compiled
against an old client shape (fields/models that the current schema no longer
has), so `tsc` either failed against code matching the *new* schema, or passed
against code matching the *old* one. The shared client is regenerated by the
Investor/Contractor portal dev workflows (`db:generate`), so its shape depends on
whenever those last ran.

**How to apply:** If `tsc` complains a Prisma field/model is missing or
unexpectedly required, regenerate the client from the authoritative schema, then
re-run `tsc`. Regeneration can *surface* breakage in other features whose code
references models not present in the committed schema (e.g. a half-merged feature
whose models were never added) — those failures are pre-existing, not caused by
your edits.
